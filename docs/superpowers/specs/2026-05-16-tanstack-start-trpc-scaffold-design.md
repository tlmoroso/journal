# Tanstack Start + tRPC Scaffold ‚Äî Design Spec

**Date:** 2026-05-16
**Status:** Approved
**Scope:** Initial scaffold for the `journal` repository ‚Äî basic project structure, dependencies, and local dev scripts. No journaling features, database, or auth in this scope.

## Goal

Stand up a working Tanstack Start application with tRPC, the broader Tanstack ecosystem (Router, Query, Table), and pnpm catalog-based dependency management. After scaffolding, `pnpm install && pnpm dev` should produce a running app with a demo route that round-trips a tRPC procedure end-to-end.

## Non-Goals

- Journaling domain features (entries, tags, editor, etc.)
- Database / ORM selection
- Authentication
- Deployment configuration
- CI configuration
- Git hooks (Husky, lint-staged) ‚Äî can be added later
- Storybook / component documentation tooling

## Architecture

### Stack

- **Frontend & server runtime:** Tanstack Start (Vite + Nitro/h3, post-Vinxi architecture from v1.121+)
- **Routing:** Tanstack Router (file-based, `src/routes/`)
- **Client data:** Tanstack Query (`@tanstack/react-query`)
- **API:** tRPC v11 mounted on a Tanstack Start API route
- **tRPC ‚Üî Query integration:** `@trpc/tanstack-react-query` (the canonical v11 integration; we do **not** use the older `@trpc/react-query`)
- **Tables:** Tanstack Table (installed; not pre-wired into a component)
- **Validation:** Zod 4
- **Language:** TypeScript (strict)
- **Package manager:** pnpm 9.5+ with catalog
- **Runtime:** Node 24 LTS

### Why Start-only (no separate Express)

Tanstack Start ships its own server runtime (Nitro/h3) and supports server-side request handlers via API routes. tRPC's `fetchRequestHandler` adapter mounts cleanly inside a Start API route, so a separate Express process is unnecessary and would only add coordination complexity (two dev servers, CORS, deploy targets). This is the canonical Tanstack Start guidance.

### Why no Vinxi

Tanstack Start removed Vinxi as of v1.121 ("devinxi" migration). Start now ships as a single Vite plugin (`tanstackStart()` from `@tanstack/react-start/plugin/vite`) plus `@vitejs/plugin-react`, with Nitro running underneath for the production server. We adopt the post-migration architecture from day one ‚Äî no Vinxi dependency, scripts run `vite` directly.

### Server boundary

- All server-only code lives under `src/server/`. Nothing under that path is imported from client modules at runtime ‚Äî only types cross the boundary (via tRPC's `AppRouter` type export).
- Tanstack Start treats files imported only from server contexts (API routes, server functions) as server-only and tree-shakes them out of the client bundle.

### Key implementation notes

- **`vite.config.ts`** ‚Äî registers two plugins:
  ```ts
  import { defineConfig } from 'vite';
  import { tanstackStart } from '@tanstack/react-start/plugin/vite';
  import viteReact from '@vitejs/plugin-react';
  export default defineConfig({
    plugins: [tanstackStart(), viteReact()],
    test: { environment: 'jsdom', setupFiles: ['./src/test/setup.ts'] },
  });
  ```
- **`src/routes/api/trpc/$.ts`** uses Tanstack Start's API route mechanism (e.g. `createServerFileRoute`) with a splat parameter to handle `/api/trpc/*`, delegating to `fetchRequestHandler` from `@trpc/server/adapters/fetch` with `appRouter` and `createContext`.
- **`src/lib/trpc.ts`** uses `@trpc/tanstack-react-query` ‚Äî exports a typed tRPC client built with `createTRPCClient<AppRouter>({ links: [httpBatchLink({ url: '/api/trpc' })] })` and the corresponding query utils for use with `useQuery`/`useMutation`. We commit to this integration; the older `@trpc/react-query` (with `createTRPCReact()`) is not used.
- **`src/router.tsx`** exports a `getRouter()` function (the canonical name Tanstack Start looks for) that creates the router with a context object holding `{ trpcClient, queryClient }`. `getRouter()` MUST return a fresh router instance per call ‚Äî singletons leak request-scoped state across SSR requests. Client and server entries are framework-managed by the `tanstackStart()` Vite plugin; the user does not author them.
- **`src/routes/__root.tsx`** wraps the app in `QueryClientProvider`, and renders `TanStackRouterDevtools` + `ReactQueryDevtools` behind `import.meta.env.DEV`.
- **`src/routes/index.tsx`** uses Tanstack Query (via the `@trpc/tanstack-react-query` utilities) to call the `health.ping` procedure and display the response, demonstrating the full client ‚Üí router ‚Üí tRPC ‚Üí server round-trip.

A single `default` catalog (no named catalogs) lives in `pnpm-workspace.yaml`. All version control flows through this one file. Named catalogs (e.g. for mixing React versions) are YAGNI for a single-package repo.

### `package.json` consumption

Every dependency in `package.json` references the catalog:

```json
{
  "dependencies": {
    "react": "catalog:",
    "@tanstack/react-start": "catalog:",
    "@trpc/server": "catalog:",
    "zod": "catalog:"
  }
}
```

This means version bumps are a one-line edit in `pnpm-workspace.yaml`.

### `engines` and `packageManager`

```json
{
  "engines": {
    "node": ">=24.0.0",
    "pnpm": ">=9.5.0"
  },
  "packageManager": "pnpm@9.x.y"
}
```

`packageManager` is set to a specific pinned pnpm version so Corepack activates the correct CLI on first install.

### Node 24 rationale

Node 24 entered Active LTS in October 2025 and remains in Active LTS through October 2026 (then Maintenance LTS until April 2028). It's the appropriate LTS choice today and gives this project ~2.5 years of security support without a forced upgrade. Node 23 was a non-LTS odd-numbered release that has reached EOL.

### Zod 4 + tRPC v11 known wrinkle

Zod v4 errors do not extend `Error`, which trips tRPC's default error formatter (it loses the validation message). The fix is a custom `errorFormatter` in `src/server/trpc/trpc.ts` that explicitly checks for `ZodError` and surfaces flattened validation issues:

```ts
import { initTRPC } from '@trpc/server';
import { z } from 'zod';
import type { Context } from './context';

const t = initTRPC.context<Context>().create({
  errorFormatter({ shape, error }) {
    const zodError = error.cause instanceof z.ZodError ? error.cause.flatten() : null;
    return { ...shape, data: { ...shape.data, zodError } };
  },
});

export const router = t.router;
export const publicProcedure = t.procedure;
```

The exact flattening API used (`z.treeifyError` vs `error.flatten()`) will use the canonical Zod 4 form at implementation time.

## Scripts & Local Dev Workflow

### `package.json#scripts`

```json
{
  "scripts": {
    "dev": "vite dev",

    "start": "node .output/server/index.mjs",

    "format:check": "prettier --check .",
    "test": "vitest run",
    "test:watch": "vitest",
    "test:ui": "vitest --ui",
    "check": "pnpm run typecheck && pnpm run lint && pnpm run format:check && pnpm run test",
    "clean": "rimraf .tanstack .output dist node_modules/.vite"
  }

```

### Onboarding flow

```bash
corepack enable          # activates the pnpm version pinned in packageManager
nvm use                  # reads .nvmrc -> Node 24
pnpm install             # resolves catalog references
pnpm dev                 # http://localhost:3000
```

The README "Getting Started" section captures this exact sequence.

### Dev server

- Single process ‚Äî Vite serves the client and Nitro/h3 serves SSR + API routes simultaneously via the `tanstackStart()` plugin.
- tRPC at `/api/trpc/*` is served by the same process; no second port to manage.
- HMR covers client routes/components and server code (`src/server/**`).
- Devtools (Tanstack Router + React Query) mount in `__root.tsx` behind `import.meta.env.DEV`.

### Production

`vite build` produces a Nitro server bundle under `.output/`. `pnpm start` runs `node .output/server/index.mjs`. Deployment target is out of scope for this spec.

### `check` script

`pnpm check` is the single canonical "is everything green?" command ‚Äî it's what CI will run. Order is intentional:

1. `typecheck` ‚Äî cheapest signal of broken code, fails fast
2. `lint` ‚Äî style/correctness rules
3. `format:check` ‚Äî formatting drift
4. `test` ‚Äî slowest, runs last

### Test setup

- `environment: 'jsdom'` for component tests (Vitest config inside `vite.config.ts`)
- `setupFiles: ['./src/test/setup.ts']` loading `@testing-library/jest-dom`
- Tests colocated next to source (`foo.ts` ‚Üî `foo.test.ts`)
- One sample test in `src/server/trpc/routers/health.test.ts` that exercises `health.ping` via a server-side tRPC caller (`appRouter.createCaller`) ‚Äî proves the stack wires up without needing a running HTTP server.
